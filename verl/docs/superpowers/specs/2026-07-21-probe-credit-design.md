# On-Policy Probe Credit Redistribution Design

## Scope

This document specifies the experimental `probe_credit` implementation for synchronous DAPO/GRPO training. The implementation measures prefix recoverability with Immediate Answer Prefix Probes generated by the current rollout policy and redistributes the existing terminal GRPO advantage across response tokens.

The feature does not change terminal verifier rewards, `token_level_scores`, `token_level_rewards`, Dynamic Sampling criteria, PPO clipping, KL handling, or the shared `RayPPOTrainer` training loop. It does not train a recoverability critic, alter existing formal offline Probe data, run an H100 job, or merge the feature branch.

## Upstream DAPO Source

The experimental trainer is based on:

- Repository: `https://github.com/verl-project/verl-recipe`
- File: `dapo/dapo_ray_trainer.py`
- Source commit: `e477deff97b15d067f8b4e71f75b80ae58ad64c4` (2026-06-09)
- Last substantive current-main DAPO adaptation in that file: `8f9e7ac93acac57f6ef7e8dffa3de788d148a09a` (2026-02-28)

The Dynamic Sampling block is migrated without changing its selection rule:

1. Generate `rollout.n` trajectories per prompt.
2. Compute terminal verifier/reward results.
3. Read `algorithm.filter_groups.metric`.
4. Keep prompt groups whose metric standard deviation is positive, plus the upstream single-trajectory compatibility case.
5. Accumulate retained prompt groups across generation batches.
6. Continue generation while fewer than `data.train_batch_size` prompts have been retained.
7. Preserve upstream `algorithm.filter_groups.max_num_gen_batches` behavior and error when the configured limit is reached.
8. Select exactly `data.train_batch_size` complete prompt groups, retaining all `rollout.n` trajectories in each group.

Compatibility changes are limited to current repository APIs, including the current checkpoint-manager signatures, rollout manager, reward extraction, debug metrics, profiling, shutdown behavior, and `TaskRunner` construction. They must not change the filtering criterion or accumulation semantics. The implementation records the source and each compatibility change in code comments and the algorithm documentation.

## Experimental Architecture

The feature uses a dedicated experimental entry point and trainer:

- `RayDAPOProbeCreditTrainer(RayPPOTrainer)` owns the migrated DAPO loop.
- A dedicated `TaskRunner` selects the experimental trainer.
- The ordinary `RayPPOTrainer.fit()` remains unchanged.
- A typed `ProbeCreditConfig` is added to `AlgoConfig`, with `enable=false` and `coef=0.0` defaults.
- A pure Dynamic Sampling helper expresses the upstream group-filtering and complete-group truncation behavior for CPU parity tests.
- A pure Probe credit module computes temporal returns, group-relative advantages, token corrections, and final advantages.
- A lightweight Probe runtime module builds raw-token requests, calls the existing vLLM rollout servers, scores candidates, aggregates branches, attaches tensors to `DataProto`, and emits metrics.
- The existing offline CLI is not a runtime dependency. Parity tests compare the lightweight implementation with the current offline Immediate Answer Probe protocol without rewriting the user's existing offline work.

The first version supports only synchronous training and the vLLM rollout backend. Configuration validation fails clearly for a fully asynchronous policy/trainer or any non-vLLM backend.

## Exact Training Flow and Policy Consistency

The experimental training step is:

```text
pi_old generates all candidate rollout batches
-> terminal verifier/reward for each candidate batch
-> official DAPO filter_groups filtering
-> accumulate retained complete prompt groups
-> select the final fixed-size retained training batch
-> pi_old generates Immediate Answer Prefix Probes for that batch only
-> put rollout replicas to sleep
-> old/ref log probabilities and optional values
-> standard GRPO terminal advantage
-> Probe Credit Redistribution
-> actor update
-> publish pi_new weights to rollout replicas
```

All Dynamic Sampling generation batches, retained trajectories, and Probe requests carry the same `policy_version`, defined as the current `global_step` before the actor update. The trainer validates this binding before probing. The first version rejects asynchronous/stale-policy execution instead of attempting to reconcile policy versions.

Probe generation is never invoked while Dynamic Sampling is accumulating batches. A trajectory that is filtered out or excluded by final complete-group selection produces zero Probe requests.

## Deterministic Probe Randomness

Probe sampling uses a separate deterministic seed namespace. Each logical prefix request receives one grouped-request seed derived from:

```text
(global_step, uid, trajectory_id, relative_position, ordered_branch_ids)
```

The derivation uses a stable hash, not Python's process-randomized `hash()`. For the prompt-level `h=0` request, `trajectory_id` is a documented prompt-level sentinel. vLLM grouped `n=4` sampling accepts one request seed, so the four outputs are reproducible samples from one isolated request RNG stream; they are not four independently seeded requests. Stable vLLM output index `b` maps to logical branch ID `b`. Probe generation must not draw from or mutate the normal rollout's shared Python, NumPy, Torch, vLLM, or backend RNG streams. Reordering prefix requests therefore does not change a grouped-request seed, branch mapping, or aggregation result.

## Immediate Answer Prefix Probe Protocol

For a retained trajectory with prompt token IDs `p`, response token IDs `y`, and absolute horizon `h`, the request input is exactly:

```text
p + y[:h] + encode("\n\nAnswer:", add_special_tokens=false)
```

The runtime operates directly on the original token IDs. It must not decode and re-encode the prompt/response prefix, apply a chat template, create a new user turn, append the response suffix, or manually append EOS.

The default grouped sampling parameters are:

```text
n=4
temperature=0.7
top_p=0.95
top_k=-1
max_tokens=64
stop=["\n"]
```

EOS and native stop behavior remain enabled. The vLLM adapter uses one prefix request with grouped `n=4` sampling, rather than four duplicate prefix-prefill requests. The grouped request has one independently derived seed, and stable output indices identify its four logical branches.

The candidate is the first non-empty generated line. The verifier input is `Answer: {candidate}` and the configured math verifier determines binary correctness. Any context overflow, missing output, invalid branch, scoring exception, or incomplete branch set raises in default `strict=true` mode. Prefixes are never silently truncated and missing values never become zero recoverability.

## Probe Positions and Deduplication

The canonical relative positions are:

```text
[0.0, 0.25, 0.50, 0.75, 0.90]
```

For response length `L`, each absolute horizon is `floor(q * L)` and must satisfy `0 <= h < L`.

- `h=0` is generated once per unique retained prompt when `probe_zero_position=true` and broadcast to all retained trajectories for that prompt.
- Identical absolute prefixes created by duplicate horizons on a short trajectory are generated once and reused at every matching relative position.
- Requests and responses use explicit IDs containing policy version, UID, trajectory identity, position, horizon, and branch identity. Aggregation never assumes return order.
- `probe_values`, `probe_relative_positions`, `probe_absolute_horizons`, and `probe_valid_mask` are attached to the final retained `DataProto` batch through a uniform downstream interface.

Recoverability is the mean of the `n` binary branch successes:

```math
V_i(h) = \frac{1}{n}\sum_{b=1}^n \mathbf{1}[\mathrm{probe}_{i,h,b}\text{ is correct}].
```

## Probe Temporal Credit

For `V_i = [V_{i,0}, ..., V_{i,4}]`, local progress is:

```math
r^P_{i,k} = V_{i,k+1} - V_{i,k}, \qquad k \in \{0,1,2,3\}.
```

Discounted backward propagation is:

```math
G^P_{i,k} = \sum_{j=k}^{3}\rho^{j-k}r^P_{i,j}, \qquad \rho \in [0,1].
```

For each prompt group and segment, the group-relative Probe advantage is:

```math
A^P_{i,k} = \frac{G^P_{i,k}-\mu_{g,k}}{\sigma_{g,k}+\epsilon}
```

when `norm_probe_by_std=true`, and `G^P_{i,k}-\mu_{g,k}` otherwise. Mean and standard deviation must use the exact convention already used by GRPO in this repository. The implementation calls `verl.utils.group_mean_std` (or the same public helper used by the active GRPO path), including its Bessel-corrected sample standard deviation, rather than introducing a population-standard-deviation convention. A group-segment with group size at most one or standard deviation at most `epsilon` contributes exactly zero. Numerical parity tests cover group sizes 2 and 8 and a degenerate group.

The prompt-shared `V(0)` term commonly cancels during group centering because every retained trajectory in that prompt receives the same value. The first version nevertheless keeps `probe_zero_position=true` by default for complete recoverability monitoring and protocol consistency. A later experiment may disable it as an efficiency optimization, provided the resulting credit definition and monitoring difference are explicit.

## Token Mapping and Zero-Mean Redistribution

For trajectory length `L`, token boundaries are:

```text
b0=0
b1=floor(0.25 L)
b2=floor(0.50 L)
b3=floor(0.75 L)
b4=floor(0.90 L)
b5=L
```

Segments `[b0,b1)`, `[b1,b2)`, `[b2,b3)`, and `[b3,b4)` receive `A^P_{i,0..3}`. The `[b4,b5)` tail receives exactly zero direct Probe correction.

The length-weighted mean over covered tokens is:

```math
\bar A_i^P = \frac{\sum_{k=0}^{3}(b_{k+1}-b_k)A^P_{i,k}}
{\sum_{k=0}^{3}(b_{k+1}-b_k)}.
```

The centered segment correction is:

```math
\widetilde A^P_{i,k}=A^P_{i,k}-\bar A_i^P.
```

Before use, the trainer verifies for every trajectory that `sum(response_mask * probe_correction) == 0` within numerical tolerance. Response padding and the 90%-100% tail must be exactly zero, not merely close to zero. This is **raw masked advantage-mass conservation**: it also preserves the unweighted sum of token advantages before the policy-loss transformation. It does not claim that PPO importance ratios, clipping, token weighting, minibatching, or configured loss aggregation preserve the final policy-gradient contribution or optimized objective exactly.

## Final Advantage Semantics

The standard GRPO result is preserved in `terminal_advantages` before modification. The implementation then sets:

```math
A^{new}_{i,t}=A^{terminal}_{i,t}+\alpha C^P_{i,t},
```

where `alpha = algorithm.probe_credit.coef`.

The implementation never changes terminal verifier output, `token_level_scores`, or `token_level_rewards`. In the current GRPO interface, `returns` is equal to `advantages`; therefore both tensors are set to the redistributed result after `terminal_advantages` is saved. This preserves the existing GRPO consumer contract and does not reinterpret Probe values as rewards.

When `enable=false`, the trainer does not build Probe requests, does not consume Probe seeds, and leaves standard GRPO outputs untouched. When `coef=0`, available Probe data produces no numerical advantage change.

## Configuration

`algorithm.probe_credit` is typed and defaults to:

```yaml
enable: false
coef: 0.0
rho: 0.5
relative_positions: [0.0, 0.25, 0.50, 0.75, 0.90]
n: 4
temperature: 0.7
top_p: 0.95
top_k: -1
max_tokens: 64
stop: ["\n"]
answer_prefix: "\n\nAnswer:"
norm_probe_by_std: true
epsilon: 1.0e-6
probe_zero_position: true
strict: true
debug_dump: false
```

Validation requires sorted positions, first position zero, all positions below one, `rho` in `[0,1]`, nonnegative `coef`, positive `n`, positive `max_tokens`, GRPO, synchronous training, and vLLM. Enabling Probe with a nonpositive coefficient is rejected by the experimental launcher so an experiment cannot silently run without redistribution. Dynamic Sampling and Probe configuration remain independent: Probe configuration cannot affect `filter_groups` decisions, and `filter_groups` configuration cannot alter Probe math.

## Metrics and Debugging

Metrics cover configuration, request counts, input/output token counts, completion length, max-token rate, error rate, position-wise recoverability, recoverability deltas, Probe returns, degeneracy, correction magnitude, zero-mean residual, and generation/scoring/credit timing.

Additional comparison metrics record the mean and standard deviation of:

- the preserved terminal advantage;
- the Probe correction;
- the final redistributed advantage.

Normal rollout wall time, Probe request construction, total Probe generation, Probe input tokens, Probe output tokens, scoring, advantage computation, actor update, and total step wall time remain distinguishable. If the backend cannot separate prefill and decode wall time, token counts and total Probe generation time provide the first-version proxy.

`debug_dump=true` writes at most eight samples per step containing UID, policy version, response length, boundaries, values, pseudo-rewards, discounted returns, normalized Probe advantages, centered corrections, terminal advantage, and final advantage summaries. Full responses are not dumped by default.

## Tests and Scientific Invariants

Pure unit tests cover boundary floor behavior, duplicate horizons, aggregation, the exact `V=[0,0,0,0.75,0.75]` example, `rho` endpoints, negative progress, group normalization, degeneracy, token mapping, zero mean, tail/padding zeros, final advantage arithmetic, disabled and zero-coefficient behavior, invalid configuration, strict failures, shuffled results, seed stability, and parity with the offline protocol.

CPU/mock integration tests cover:

- official Dynamic Sampling filtering and accumulation parity;
- complete prompt-group truncation;
- identical retained IDs with Probe disabled or enabled;
- zero Probe requests for filtered and final-truncation trajectories;
- requests only after the final retained batch exists;
- policy-version equality across generation batches and Probe requests;
- Probe completion before actor update;
- raw-token prefix construction without decode/re-encode;
- h0 and duplicate-prefix deduplication;
- grouped branch aggregation under shuffled return order;
- context-overflow fail-fast behavior;
- unchanged terminal rewards and token-level rewards;
- per-trajectory zero-sum correction, exact padding/tail zeros, unchanged tensor shapes, and terminal-advantage anchoring.

The required invariants are:

1. Feature-off behavior is the migrated official DAPO baseline.
2. Terminal reward and Dynamic Sampling criteria are unchanged.
3. All candidate batches and Probes use one synchronous pre-update policy version.
4. Probe RNG is isolated from normal rollout RNG.
5. Probe requests exist only for complete final retained groups.
6. Probe failures cannot silently become zero rewards.
7. Probe correction provides raw masked advantage-mass conservation within each trajectory; it makes no claim of invariant post-ratio, post-clipping, or post-aggregation gradients.
8. Padding and the 90%-100% tail have exact zero correction.
9. Probe finishes before any actor update for the batch.
10. Enabling Probe does not change retained prompt or trajectory identities.

## Delivery

The feature is delivered on `agent/onpolicy-probe-credit` with code, typed configuration, a dedicated experimental launcher, documentation, unit tests, CPU/mock integration tests, a reviewed commit, and a push to the feature branch. No GPU allocation, Slurm submission, formal DAPO training, main-branch merge, or destructive repository operation is part of this work.
